### Issues Found

**❌ `app/layout.tsx:24` — Hardcoded `lang="en"` on root layout**

```tsx
// app/layout.tsx lines 22-35
<html
  lang="en"              // ❌ Should not be here
  suppressHydrationWarning // ❌ Masks hydration bugs
>
```

The root layout (`app/layout.tsx`) should NOT set `lang` since it's locale-unaware. The `lang` attribute belongs only in `app/[locale]/layout.tsx` where the locale is known.

**Fix:** Move `<html lang={locale}>` to `app/[locale]/layout.tsx`. The root layout should either:

- Not render `<html>` at all (let the locale layout handle it), or
- Render a bare `<html>` without `lang`

**❌ `app/layout.tsx:25` — `suppressHydrationWarning` used as workaround**

This attribute suppresses React hydration mismatch warnings. When used on `<html>`, it likely hides i18n-related mismatches caused by server/client locale differences. This is a band-aid, not a fix.

  const requested = await requestLocale
  const locale = hasLocale(routing.locales, requested)

| Check                                            | Result |

### Issues Found

**❌ No `timeZone` configured**

Missing from the return object:

```ts
return {
  locale,
  timeZone: "Europe/Paris", // or 'America/New_York', etc.
  messages: (await import(`../messages/${locale}.json`)).default,
}
```

Without `timeZone`, all date/time rendering will use the server's timezone, causing inconsistencies between server-rendered and client-rendered content.

**⚠️ No error handling for missing message files**

If `messages/xx.json` doesn't exist, the dynamic import will throw an unhandled error during build/runtime. Consider:

```ts
try {
  messages = (await import(`../messages/${locale}.json`)).default
} catch {
  messages = (await import(`../messages/${routing.defaultLocale}.json`)).default
}
```

## 7. Locale Layout

**Status: ⚠️ WARNING**

**Files checked:** `app/[locale]/layout.tsx`

```tsx
// app/[locale]/layout.tsx
export default async function LocaleLayout({
  children,
  params,
}: Readonly<{
  children: React.ReactNode
  params: Promise<{ locale: string }>
}>) {
  const { locale } = await params

  if (!hasLocale(routing.locales, locale)) {
    notFound()
  }

  setRequestLocale(locale)
  const messages = await getMessages()

  return (
    <NextIntlClientProvider messages={messages}>
      <ThemeProvider>
        <TooltipProvider>{children}</TooltipProvider>
      </ThemeProvider>
    </NextIntlClientProvider>
  )
}
```

| Check                                             | Result |
| ------------------------------------------------- | ------ |
| `params` awaited correctly                        | ✅     |
| `hasLocale` validation with `notFound()`          | ✅     |
| `setRequestLocale(locale)` called                 | ✅     |
| `generateStaticParams` exported                   | ✅     |
| `NextIntlClientProvider` wrapping children        | ✅     |
| `generateMetadata` with locale-aware translations | ✅     |

### Issues Found

**❌ No `<html lang={locale}>` in locale layout**

The locale layout does not render `<html>` at all — it delegates to the root layout which hardcodes `lang="en"`. The `lang` attribute should be set dynamically:

```tsx
// app/[locale]/layout.tsx should render:
<html lang={locale} dir={locale === 'ar' ? 'rtl' : 'ltr'}>
```

This requires restructuring so the locale layout is the one rendering `<html>` and `<body>`.

**❌ No RTL `dir` attribute support**

No logic for `dir="rtl"` if RTL languages are ever added.

**⚠️ `messages` passed manually to `NextIntlClientProvider`**

In next-intl v3+/v4, the provider auto-reads messages from server context. Passing `messages={messages}` manually is the outdated pattern. The provider should work without it:

```tsx
<NextIntlClientProvider>{children}</NextIntlClientProvider>
```

## 11. Date, Number & Pluralization Formatting

**Status: ❌ FAIL**

**Files checked:** `app/[locale]/(admin)/admin/users/page.tsx`, `app/[locale]/(admin)/admin/_components/data-table.tsx`, `app/[locale]/(admin)/admin/_components/chart-area-interactive.tsx`, `components/ui/chart.tsx`

### Issues Found

**❌ `toLocaleDateString()` used without locale (4 locations)**

```tsx
// app/[locale]/(admin)/admin/users/page.tsx:383
{
  new Date(user.createdAt).toLocaleDateString()
}

// app/[locale]/(admin)/admin/users/page.tsx:580
{
  new Date(session.createdAt).toLocaleString()
}

// app/[locale]/(admin)/admin/_components/data-table.tsx:149
{
  new Date(row.original.createdAt).toLocaleDateString()
}
```

These cause hydration mismatches because the server and client may have different default locales.

**❌ Hardcoded `"en-US"` locale in chart (2 locations)**

```tsx
// app/[locale]/(admin)/admin/_components/chart-area-interactive.tsx:247
return date.toLocaleDateString("en-US", {
  month: "short",
  day: "numeric",
})

// chart-area-interactive.tsx:258
return new Date(value).toLocaleDateString("en-US", {
  month: "short",
  day: "numeric",
})
```

French users will see English month abbreviations ("Apr", "May") instead of French ("avr.", "mai").

**❌ `toLocaleString()` for numbers**

```tsx
// components/ui/chart.tsx:239
{
  item.value.toLocaleString()
}
```

**Fix:** Use next-intl's `useFormatter` hook:

```tsx
const format = useFormatter()
format.dateTime(date, { month: "short", day: "numeric" })
format.number(value)
```

**No ICU plurals or rich text found in message files.** All translation values are plain strings. If plurals are ever needed, ICU syntax should be used:

```json
{ "items": "{count, plural, =0 {No items} =1 {One item} other {# items}}" }
```

## 15. TypeScript Integration

**Status: ❌ FAIL**

**Files checked:** `tsconfig.json`, glob for `*.d.ts` files

### Issues Found

**❌ No `IntlMessages` type augmentation**

Translation keys have zero type safety. A typo like `t("common.singIn")` instead of `t("common.signIn")` won't be caught by TypeScript.

**Fix:** Create a type augmentation file:

```ts
// types/next-intl.d.ts
import en from "../messages/en.json"

type Messages = typeof en

declare global {
  interface IntlMessages extends Messages {}
}
```

**❌ No exported `Locale` type from routing config**

```ts
// lib/routing.ts — add:
export type Locale = (typeof routing.locales)[number]
```

This enables type-safe locale parameters throughout the codebase.

**⚠️ `params` type could be stricter**

While `Promise<{ locale: string }>` is correct, using the exported `Locale` type would be better:

```ts
params: Promise<{ locale: Locale }>
```
